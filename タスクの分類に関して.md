# タスクの分類
## 直列必須のタスク
- 前の作業が終わらないと次に進めない作業
- 特徴
  - 明確な依存関係がある
  - 順番を守らないと手戻りが発生する
  - 1つずつしか進められない
- 例
  - 例1）要件定義→基本設計→詳細設計→実装→テスト
    - 「要件定義」が終わらないと「設計」 が始められない
  - 例2）一覧ページ→詳細ページ
    - 「一覧ページ」が完成しないと、「詳細ページ」の実装には入れない
## 並列可のタスク
- 他のタスクと同時に進めても問題ない作業
- 特徴
  - 前後関係がない
  - どれから始めてもOK
  - 複数人・複数時間帯で同時進行できる
- 例
  - 画面デザイン作成
  - DBテーブル設計
  - API仕様書作成
  → これらは互いに大きく依存していなければ、同時に進められる

## 前後関係、依存関係を見極めるポイント
### ① 「これが無いと始められないものは何か？」と自問する
- 見極めの基本質問
  - **このタスクを始めるために何が確定していないといけないか？**
- YESなら 依存あり（直列寄り） です。
- 例
  - 実装
    - → API仕様が決まってないと書けない ❌
  - 画面モック作成
    - → 多少曖昧でも作れる ⭕️
- 👉 「入力が未確定だと手が止まるか？」 が重要
### ② 成果物（アウトプット）に注目する
- タスクを「作業」ではなく成果物で見ると判断しやすい
- チェックポイント
  - **その成果物は他のタスクのインプットになっているか？**
  - Yes　→ 前工程
- 例

| タスク  | 成果物    | 依存 |
| --- | --- | ---|
| 要件定義 | 要件書    | 設計が依存(要件定義がないと設計に入れない) |
| DB設計 | テーブル定義 | 実装が依存(テーブル定義がないと実装できない) |
| テスト  | テスト結果  | リリース判断が依存<br>（テストが完了しないとリリースできない） |
- 👉 「誰が次に使う？」と考える
### ③ 変更が起きたら致命的か？
- 見極めの質問
  - **後から変更が入ったら、どれくらい壊滅的か？**
- 壊滅的 → 直列必須
- 多少の修正で済む → 並列可寄り
- 例
  - 要件変更
    - → 実装・テスト全やり直し（致命的）
  - 文言修正
    - → 表示修正だけ（軽微）
- 👉 「戻りコスト」で判断
### ④ 仮置き・仮定で進められるか？
- 見極め質問
  - **仮の前提を置いて進めても、後で吸収できるか？**
- できる → 並列可
- できない → 直列必須
- 例
  - UIデザイン
    - → 仮データで作れる ⭕️
  - DB正規化
    - → 仮でやると破綻しやすい ❌
### ⑤ 人・役割が分かれているか？
- 見極めポイント
  - 同じ人がやらないといけない？
  - 専門スキルが被っていない？
- 例
  - フロントエンド
    - UIデザイン
  - バックエンド
    - DB定義
- → APIのインターフェースだけ決めれば並列可
- 👉 境界（IF）を決められるかがカギ
### ⑥ 時間ではなく「決定」を依存関係の軸にする
- よくある誤解👇
  - ❌「時間がかかるから先にやる」
  - ⭕️「意思決定が必要だから先にやる」
- 直列必須になりやすいもの
  - 方針決定
  - ルール確定
  - 仕様確定
- **顧客への質問・確認などは最優先で行う**
- 
### ⑦ 実務で使える簡易チェックリスト
- [ ] タスクを始めるのに必要な情報は揃っているか？
- [ ] 成果物は他タスクの前提になっているか？
- [ ] 後戻りしたら被害が大きいか？
- [ ] 仮決めで進められるか？
- [ ] インターフェースを先に固定できるか？
- YESが多いほど直列必須
## まとめ
- 要件定義は必ず設計前に確認・確定させる必要がある
- 意思決定（顧客への仕様確定の確認、質問etc）が必要なタスクは最優先で実行する
